# AGENTS.md

이 저장소에서 작업할 때 지키는 규칙이다.

## 프로젝트 개요

- **과제**: 커피숍 주문 시스템 (K사 서버 개발 사전과제)
- **핵심 기능**: 메뉴 조회, 포인트 충전, 주문/결제, 인기 메뉴 조회
- **핵심 설계 관점**: 동시성 제어, 데이터 일관성, 다수 서버 인스턴스 대응
- **스택**: Java 21, Spring Boot, Gradle, JPA

## 작업 단계

### 1단계 — 무엇을 만들지 확인
- 이번 작업이 다루는 API를 `docs/api/{도메인}-api.md`에서 확인한다. 없으면 먼저 작성하고 검토를 받는다. (`docs/workflow/plan-guide.md` 1~2단계)

### 2단계 — 테이블 확인
- 필요한 테이블을 `docs/db/erd.md`에서 확인한다. 없거나 수정이 필요하면 반영 후 검토를 받는다. (`docs/workflow/plan-guide.md` 3~4단계)

### 3단계 — 코드 작성
- `docs/convention.md`의 규칙(패키지 구조, Entity/DTO/Repository/Service/Controller 작성 규칙, 네이밍, 공통 응답 형식)을 따라 작성한다.
- 1단계에서 확인한 API 명세의 요청/응답 필드, 에러 코드를 그대로 따른다.
- 동시성·트랜잭션이 걸리는 로직(포인트 차감, 주문 생성 등)은 어떤 전략(락 종류, 트랜잭션 범위)을 썼는지 설명한다.
- 요청하지 않은 기능·리팩토링을 임의로 추가하지 않는다. 범위를 벗어나야 하는 상황이 생기면 먼저 알리고 사람의 확인을 받는다.

### 4단계 — 검증
- `./gradlew test`를 실행하고 결과를 사실대로 보고한다 (통과/실패 개수, 실패 원인).
- API 명세와 실제 구현이 일치하는지, 동시성이 걸린 로직의 트랜잭션 경계가 올바른지 확인한다.
- 실패하면 원인을 설명하고 2~3단계로 돌아가 다시 시도한다.

### 5단계 — 기록
- 이번 작업의 시도·결과·API 샘플을 `docs/logs/{도메인}/*.md`에 남긴다. 방법은 `docs/workflow/logs-guide.md`를 따른다.

## 작업 모드

- 기본 모드: 1~2단계는 사람의 명시적 승인 없이는 3단계(코드 작성)로 넘어가지 않는다.
- 완전 자율 모드로 전환하라는 별도 지시가 없는 한 위 승인 절차를 항상 지킨다.
