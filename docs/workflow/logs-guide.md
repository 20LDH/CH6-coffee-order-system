# 개발 로그 가이드 (logs-guide)

개발 **실행 이력·증거**를 `docs/logs/`에 남기는 방법이다.
로그는 두 역할을 한다:
1. **피드백 센서** — 에이전트가 이걸 읽고 다음 시도(루프)를 결정한다.
2. **증거·감사** — 사람이 "실제로 이렇게 돌아갔다"를 확인한다.

> 다른 문서와의 구분: `docs/api/*.md`는 **스펙(계획·목표 형태)**, `docs/logs/`는 **실행 증거(raw 이력)**.
> 서로 안 겹친다.

## 위치 & 이름

```
docs/logs/{도메인}/00X-{작업}.md      예: docs/logs/menu/001-menu-list.md
```

- 도메인은 패키지 구조와 동일하게 맞춘다 (`menu`, `point`, `order`).
- API 문서(`docs/api/{도메인}.md`)와 **같은 도메인 이름**으로 대응시킨다 (`docs/api/menu.md` ↔ `docs/logs/menu/…`).

## 무엇을 기록하나

- **성공·실패 매 시도를 모두** 기록한다 (append-only). 실패를 빼면 에이전트가 같은 실수를 반복한다.
- 각 기록은 **고신호 요약**으로 남긴다. 원시 로그 통째 덤프 금지 (노이즈 → 판단 방해 + 컨텍스트 낭비).
- **무거운 산출물**(테스트 HTML 리포트)은 `build/`에 두고, 로그엔 **요약 + 경로**만.

### 기록해야 할 증거

- ⭐ **API 요청/응답 샘플** — "엔드포인트가 실제로 이렇게 응답" (가장 핵심 증거)
- **테스트 결과 요약** — 통과/실패 개수 + 실패 원인
- **동시성/트랜잭션 관련 결정** — 포인트 차감, 주문 생성처럼 락·트랜잭션 경계가 걸린 로직은 왜 그렇게 설계했는지

## 로그 템플릿 (append-only)

```markdown
# 001-menu-list — 메뉴 목록 조회 (로그)

## Attempt 1 — 2026-07-15  ❌ FAIL
- 시도: GET /api/menus 구현 (Menu, MenuRepository, MenuService, MenuController)
- 결과: `MenuControllerTest` 실패
- 원인: 응답 필드명이 docs/api/menu.md 스펙과 불일치 (`id` → `menuId`여야 함)
- 다음: MenuResponse 필드명 수정

## Attempt 2 — 2026-07-15  ✅ PASS
- 시도: MenuResponse 필드명을 menuId로 수정
- 결과: `./gradlew test` 전체 통과
- 증거(API 샘플):
  - `GET /api/menus` → `200 {"status":200,"message":"요청이 성공했습니다.","data":{"menus":[{"menuId":1,"name":"아메리카노","price":4000}]}}`
```

## 누가 기록하나

한 Attempt는 두 부분이고, 각 단계가 **자기가 아는 부분**을 기록한다 (순차 실행이라 충돌 없음):

- **Generate 단계**: `시도` — 무엇을·어떤 접근으로 구현했는가 (자기가 한 일을 firsthand로 기록).
- **Evaluate 단계**: `결과`·`원인`·`증거` — 테스트 통과/실패, API 샘플 (실행 결과를 append).

## 워크플로우 연결

- 재시도(루프) 시 **다음 시도는 fresh 컨텍스트**일 수 있어 이전 시도를 모른다 →
  **로그를 먼저 읽어** 같은 접근을 반복하지 않는다. (그래서 Generate 단계가 자기 `시도`를 남겨야 한다.)
- (지속 기억: 세션이 바뀌거나 컨텍스트가 압축돼도 이 로그가 시도 이력을 보존한다.)
