# 계획(설계) 작성 가이드 (plan-guide)

"OO API를 만들자" 같은 작업을 지시받았을 때 코드를 작성하기 전에 거쳐야 하는 단계를 정리한다.

## 파일 이름 짓기 규칙

- 계획 문서 파일명: `{작업-내용}-plan.md`
    - 작업 내용은 영문 kebab-case로 무엇을 하는 작업인지 알 수 있게 짓는다.
    - 예: 상품 등록 API → `product-create-plan.md`
    - 예: 주문/결제 API → `order-payment-plan.md`

## 파일 저장 위치

| 산출물 | 저장 위치 |
|---|---|
| API 명세 | `docs/api/{도메인}-api.md` |
| 테이블(엔티티) 명세 | `docs/db/erd.md` (전체 테이블을 한 파일에 통합 관리) |

## 설계 단계
### 1단계 — API 명세 작성
- 요청받은 기능의 API 명세(Endpoint, Request/Response, Error Case)를 작성한다.
- `docs/api/`에 저장한다.

### 2단계 — API 명세 검토
- 작성한 API 명세를 보여주고 검토·승인을 받는다. 승인 전에는 다음 단계로 넘어가지 않는다.

### 3단계 — 테이블 명세 반영
- API 기능에 필요한 테이블(컬럼/타입/제약조건/관계)을 `docs/db/erd.md`에 추가·수정한다.

### 4단계 — 테이블 명세 검토
- 수정한 `docs/db/erd.md`를 보여주고 검토·승인을 받는다. 승인 전에는 다음 단계로 넘어가지 않는다.

### 5단계 — 구현
- 승인된 API 명세(`docs/api/*.md`) + 테이블 명세(`docs/db/erd.md`)를 기반으로 바로 코드를 작성한다.
- 코드 작성 규칙(패키지 구조, 계층별 작성 규칙, 네이밍 등)은 `docs/convention.md`를 따른다.
- 동시성/트랜잭션이 걸리는 로직(락 전략, 트랜잭션 범위)은 구현하면서 근거와 함께 설명하고, 사람의 확인을 받는다.

## 설계 이후
- 계획이 승인되면 승인된 범위 안에서만 구현을 진행한다.
- 구현 후 테스트 등으로 검증하고 실패 시 다시 설계 단계로 돌아간다.
