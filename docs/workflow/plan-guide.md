# 계획(설계) 작성 가이드 (plan-guide)

"OO API를 만들자" 같은 작업을 지시받았을 때 코드를 작성하기 전에 거쳐야 하는 단계를 정리한다.

## 파일 이름 짓기 규칙

- 계획 문서 파일명: `{작업-내용}-plan.md`
    - 작업 내용은 영문 kebab-case로 무엇을 하는 작업인지 알 수 있게 짓는다.
    - 예: 상품 등록 API → `product-create-plan.md`
    - 예: 주문/결제 API → `order-payment-plan.md`

## 파일 저장 위치

| 산출물 | 저장 위치 |
|---|---|
| API 명세 | `docs/api/{도메인}-api.md` |
| 테이블(엔티티) 명세 | `docs/db/{도메인}-table.md` |
| 계획 문서 | `docs/dev/plan/{작업-내용}-plan.md` |

## 설계 단계

### 1단계 — API 명세 작성
- 요청받은 기능의 API 명세(Endpoint, Request/Response, Error Case)를 작성한다.
- `docs/api/`에 저장한다.

### 2단계 — API 명세 검토
- 작성한 API 명세를 사람에게 보여주고 검토·승인을 받는다. 승인 전에는 다음 단계로 넘어가지 않는다.

### 3단계 — 테이블 명세서 작성
- API 기능에 필요한 테이블(컬럼/타입/제약조건/관계)을 정의한다.
- `docs/db/`에 저장한다.

### 4단계 — 테이블 명세 검토
- 작성한 테이블 명세를 사람에게 보여주고 검토·승인을 받는다. 승인 전에는 다음 단계로 넘어가지 않는다.

### 5단계 — 상세 구현 계획 작성
- 승인된 API 명세 + 테이블 명세를 기반으로 상세 구현 계획을 세운다.
    - API 관련 작업(Controller/Service/DTO)은 API 명세 기준.
    - Entity 관련 작업은 테이블 명세 기준.
    - 동시성/트랜잭션이 걸리는 로직이면 락 전략·트랜잭션 범위도 이 단계에서 정한다.

### 6단계 — 계획서 저장
- 위 상세 계획을 `docs/dev/plan/{작업-내용}-plan.md`로 저장한다.

### 7단계 — 계획 검토
- 저장한 계획 문서를 사람에게 보여주고 검토·승인을 받는다. 승인 전에는 코드 작성(Generate)으로 넘어가지 않는다.

## 설계 이후
- 계획이 승인되면 승인된 범위 안에서만 구현을 진행한다.
- 구현 후 테스트 등으로 검증하고 실패 시 다시 설계 단계로 돌아간다.
